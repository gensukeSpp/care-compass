## 目的

すでにログイン機能（Auth）がある前提で、**「誰がどのボードを見られるか」**を厳密に制御します。

## 1. データベース設計（Schema）

まず、SupabaseのSQL Editorで以下のテーブル構造を作成します。

### テーブル構成図

- **`profiles` テーブル**（対象者の情報）
    - `id`: uuid (Primary Key)
    - `name`: text (例: "父", "祖母")
    - `created_by`: uuid (作成した家族のID)
        
- **`board_members` テーブル**（閲覧権限の管理）
    - `profile_id`: uuid (Foreign Key -> profiles.id)    
    - `user_id`: uuid (Foreign Key -> auth.users.id)    
    - `role`: text ('owner' または 'member')
        
- **`sticky_notes` テーブル**（既存の付箋）
    - `profile_id`: uuid (Foreign Key -> profiles.id)    
    - ※これまでのデータ構造に `profile_id` を追加します。

## 2. RLS（行単位セキュリティ）ポリシーの設定

Supabaseの最大の肝です。クライアント側（React）でどう書こうが、**DB側で「権限のないデータは返さない」**設定を強制します。

### 設定するポリシーのロジック

1. **Profilesへのアクセス**: `board_members` に自分の `user_id` が存在する `profile_id` のみ取得可能。
2. **Sticky_notesへのアクセス**: その付箋が属する `profile_id` が、自分の参加しているボードのものである場合のみ取得可能。

**SQLコード例（Supabase RLS）:**

SQL
```
-- profiles テーブル用
CREATE POLICY "自分がメンバーであるプロファイルのみ表示" ON profiles
  FOR SELECT
  USING (
    id IN (
      SELECT profile_id FROM board_members 
      WHERE user_id = auth.uid()
    )
  );
```

```
-- sticky_notes テーブルにRLSを適用
ALTER TABLE sticky_notes ENABLE ROW LEVEL SECURITY;

-- 閲覧・編集ポリシー
CREATE POLICY "家族のみアクセス可能" ON sticky_notes
  FOR ALL -- SELECT, INSERT, UPDATE, DELETE すべてに適用
  USING (
    profile_id IN (
      SELECT profile_id FROM board_members 
      WHERE user_id = auth.uid()
    )
  );
```

## 3. 実装・実行プラン

### Step 1: ボード作成時の処理（オーナー登録）

家族の代表（例：長男）が新しい対象者を作成した際、以下の処理を同一トランザクションで行います。

1. `profiles` に新規レコード作成。
2. `board_members` に `user_id: 自分のID`, `role: 'owner'` でレコード作成。

### Step 2: 招待機能の実装

他の家族を追加する場合、その人の `user_id`（またはメールアドレスから取得したID）を `board_members` に追加する画面を作成します。

### Step 3: React側のデータ取得コードの変更

これまでの「すべての付箋を出す」ロジックに、`profile_id` によるフィルタを追加します。

TypeScript

```
// Supabaseから特定の対象者の付箋だけを取得する
const { data, error } = await supabase
  .from('sticky_notes')
  .select('*')
  .eq('profile_id', currentProfileId); // 現在選択中の対象者IDで絞り込み
```

## 4. セキュリティ要件のチェックリスト

|**項目**|**内容**|
|---|---|
|**なりすまし防止**|`auth.uid()` を使用し、リクエストを送ってきた本人以外のデータは絶対に返さない。|
|**誤操作防止**|`role` が 'member' のユーザーには、ボード全体の削除権限を与えないポリシーを設定。|
|**データの独立性**|`profile_id` をすべてのテーブルの主軸にし、異なる対象者のデータが混ざらないようにする。|
