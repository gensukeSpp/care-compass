## 1. アプリのコンセプト

- **アプリ名（仮）：** ケア・コンパス（Care Compass）  
- **解決したい課題：** * 高齢者の「本当はできること」を奪わず、かつ「危険なこと」を見逃さない適切な距離感の把握。    
    - 家族間での「何を手伝えばいいか」という認識のズレの解消。    
- **ターゲット層：** 軽度要介護・要支援の高齢者、およびその家族。   
- **一言で言うと：** 「本人の自立と安全を可視化し、家族のサポートを最適化する4象限ボード」  

## 2. 主要機能の一覧（ケアプラン特化版）

### 【必須機能 (MVP)】

- **4象限ボード（生活機能マップ）：**  
    - **できる：** 本人の自信維持。       
    - **できない：** サービスや家族の介入が必要。       
    - **危険を伴う：** 見守りや環境改善が必要。       
    - **頼みたい：** 本人の意思を尊重する項目。      
- **Markdown入力：** ケアマネジャーの報告書などをそのままコピペして付箋化。   
- **Google Keep連携：** 日々の気づき（「今日、階段でふらついていた」など）をKeepにメモしておき、後でボードに整理。  

### 【拡張・希望機能（色分けと分類）】

ご提案の「科目」をタグとして実装し、色分けします。

- **色分けカテゴリ例：**   
    - 🟦 **居住・環境：** 段差、手すり、掃除、火の始末       
    - 🟧 **食生活：** 調理、買い物、栄養管理、嚥下
    - 🟩 **身体・体力：** 歩行、入浴、排泄、リハビリ      
    - 🟥 **医療・健康：** 服薬管理、持病の記録、通院     
    - 🟪 **社会・交流：** デイサービス、趣味、近所付き合い       

## 3. 画面遷移図（UIイメージ）

1. **ダッシュボード（ボード一覧）：** 対象者（父、母など）を選択。   
2. **メインボード画面：** 4分割画面。右側に「Google Keep/保留ボックス」の引き出し。   
3. **付箋詳細（モーダル）：** * タイトル、詳細説明（Markdown形式）、カテゴリ、最終更新日。 
    - 「危険を伴う」から「できない」へ移動した履歴などの簡易ログ。      

## 4. 実装で望んでいる「拡張性」への提案

Reactでの実装において、将来的に以下の展開に耐えられる設計を推奨します。

- **コンポーネントの抽象化：** * 将来「4分割」だけでなく、別の軸（例：緊急度×重要度）でも表示できるように、ボードの枠組みとコンテンツ（付箋）を切り離して設計します。 
- **データ構造：**   
    - 各付箋（Note）に `category` や `status`（どの象限か）を持たせ、将来的には「時系列での変化（以前はできたが、今は危険など）」をグラフ化できるようなスキーマにします。     
- **マルチデバイス対応：**    
    - 家族が外出先でも確認できるよう、モバイルフレンドリーなCSS（Tailwind）と、ピンチイン・アウトの操作性を重視します。

## アーキテクチャ

### 1. MVPのゴール（初期開発の目標）

- **4象限ボードの表示**: 固定の十字線で区切られた1画面UI。
- **付箋の基本操作**:
    - フォームからの入力（タイトル・Markdown詳細・カテゴリ）。        
    - ドラッグ＆ドロップによる自由な配置（**絶対座標保持**）。    
- **詳細表示**: 付箋をクリックすると、Markdownがレンダリングされた詳細パネルが開く。
- **データ保存**: ブラウザの `LocalStorage` を使い、リロードしても消えないようにする。

### 2. 「拡張性」のための設計指針

後からの機能追加（家族共有、境界線可変、履歴管理）をスムーズにするために、以下の3点を意識してコードを書くことをお勧めします。

#### ① データ構造の抽象化（将来のDB移行を見据えて）

付箋のデータに「座標」と「象限ID」の両方を持たせます。

```json
{
  id: "uuid-001",
  title: "買い物（重いもの）",
  content: "お米やペットボトルの水など。週に一度は頼みたい。",
  category: "food",      // 色分け用
  status: "request",     // どの象限か（できる/できない/危険/頼みたい）
  position: { x: 120, y: 250 }, // 象限内の絶対座標、または全体ボードの座標
  author: "Me",          // 将来的にユーザー名を入れる場所
  updatedAt: "2024-05-20T..."
}
```

#### ② 状態管理の分離（Zustand の推奨）

Reactの標準の `useState` だけで管理すると、付箋が増えた時にコードが複雑になります。`Zustand` などの軽量な状態管理ライブラリを使うと、**「データの操作（移動、追記、削除）」と「見た目（Reactコンポーネント）」を完全に切り離せる**ため、後から共有機能（Supabase等）を追加する際、データ操作部分を差し替えるだけで済みます。

#### ③ 座標計算の独立

「境界線の可変」を後から入れるため、付箋の座標は「画面左上からのpx」ではなく、**「ボード全体の何％の位置にあるか」**という比率で持っておくか、あるいは座標計算ロジックを一箇所にまとめておくと、枠のサイズが変わった際に対応しやすくなります。

---

### 3. おすすめの技術選定（MVP開発用）

- **Frontend**: React (Vite) + TypeScript  
- **Styling**: Tailwind CSS（レイアウト調整が容易）  
- **D&D Library**: `dnd-kit`   
    - 非常に拡張性が高く、絶対座標の保持や、後からの「境界線を越えた時の挙動」のカスタマイズがしやすいです。     
- **Markdown**: `react-markdown`  
- **Icon**: `Lucide React`（介護・生活系のアイコンが豊富）

## 推奨フォルダ構成

Plaintext

```
src/
├── assets/             # 画像、アイコン、グローバルCSS
├── components/         # 再利用可能なUI部品
│   ├── common/         # ボタン、入力フォーム、モーダル（shadcn/ui等）
│   ├── board/          # ボード本体、十字線、象限エリア
│   └── sticky-note/    # 付箋本体、詳細表示、Markdownレンダラー
├── hooks/              # カスタムフック（D&Dロジック、LocalStorage操作）
├── lib/                # 外部ライブラリ設定（supabase.ts, markdown-config.ts）
├── services/           # 外部API連携（Google Keep API, DB操作）
├── store/              # 状態管理（Zustand: 付箋の位置、内容、ユーザー情報）
├── types/              # TypeScriptの型定義（StickyNote, Category, User）
├── utils/              # 共通関数（座標計算、日付フォーマット）
├── App.tsx             # メインレイアウト
└── main.tsx            # エントリーポイント
```
